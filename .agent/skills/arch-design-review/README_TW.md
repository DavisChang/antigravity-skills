# 架構設計審查 — Agent Skill

一個 Cursor Agent Skill，以結構化的方式引導系統架構設計討論。不是產出一份單向的設計文件，而是透過**蘇格拉底式對話**帶你走過**七大關鍵維度**，挖掘盲點並記錄取捨。

## 為什麼需要這個 Skill？

架構審查經常遺漏關鍵面向，因為：

- 團隊只關注「快樂路徑」，跳過故障模式
- 流量 / 容量估算被草草帶過：「之後再 scale 就好」
- 安全和合規在上線後才補
- 成本和團隊能力很少在為時已晚之前被討論
- 決策口頭說了就算，從未被記錄

這個 Skill 確保每個維度至少被碰過一次，關鍵維度則深入探討。

## 七大維度

| # | 維度 | 涵蓋範圍 |
|---|------|---------|
| ① | **需求與成功定義** | 商業目標、範圍/非目標、使用者旅程、功能清單 |
| ② | **流量、性能與容量** | 流量模型、延遲預算、讀寫型態、成長曲線 |
| ③ | **資料與一致性** | 一致性需求、交易、冪等、資料生命週期、資料品質 |
| ④ | **可靠性與韌性** | SLO、故障模式 (FMEA)、依賴治理、背壓、災難復原 |
| ⑤ | **安全、隱私、合規** | 威脅模型、濫用防護、機密管理、隱私、審計 |
| ⑥ | **架構形狀與可維運** | 服務邊界、同步/非同步、可觀測性、部署、測試、遷移 |
| ⑦ | **成本與組織現實** | 成本模型、供應商鎖定、團隊能力、ADR、退場 |

## 運作方式

1. **你描述要設計的系統** — 新系統、重構、或待審查的設計
2. **Agent 逐維度帶你走過** — 提出探索性問題、揭露風險、記錄決策
3. **你得到結構化產出** — 目標與 SLO、流量假設、架構草圖、Top 10 風險、取捨紀錄 (ADR)、分階段上線計畫

### 討論方式

Agent 是**思考夥伴**，不是講師：

- 用**反事實提問**測試假設（「100 倍流量時哪裡先爆？」）
- 答案含糊時**深入追問**（含糊通常是盲點的信號）
- 答案清晰合理時**記錄後繼續**
- 維度不適用時**標記 N/A**

## 輸出格式

討論後，Agent 產出結構化摘要：

```
(1) 目標與 SLO          — 3 個成功指標 + 目標值
(2) 流量假設            — 平均 / 尖峰 / 突發 / 6 個月預估
(3) 架構草圖            — 控制平面 / 資料平面 / 事件平面
(4) Top 10 風險         — 含維度、嚴重度、緩解策略
(5) 取捨 / ADR          — 3+ 個關鍵決策，含理由和推翻條件
(6) 落地計畫            — MVP → V1 → V2，標注每版解決哪些風險
```

## 使用方式

透過以下關鍵字觸發此 Skill：
- "system design"、"architecture design"、"design review"、"arch review"
- "系統架構"、"架構設計"、"架構討論"

或直接請 Agent 審查一個系統架構。

### 範例提示

```
「我正在設計一個即時通知系統，我們來做架構審查。」

「我們在重構支付服務，可以帶我走過需要考量的面向嗎？」

「審查這個架構：[貼上你的設計文件或描述系統]」
```

## 客製化

- **新增領域特定清單**：在 `dimensions.md` 中擴充你所屬產業的檢查項目（金融科技、醫療、遊戲等）
- **調整深度**：Agent 會自動適應 — 快速審查每個維度碰一次，深度審查則逐項展開
- **修改輸出格式**：調整 `SKILL.md` 中的輸出模板，以符合團隊的設計文件標準
