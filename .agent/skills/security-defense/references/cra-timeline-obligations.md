# CRA 時程與義務完整對照

> EU 2024/2847 Cyber Resilience Act — 關鍵日期與對應行動

---

## 時間軸總覽

```
2024.12.11  CRA 生效（Entry into force）
    │
    │  ← 21 個月製造商寬限期
    │
2026.06.11  Notified bodies 相關規定開始適用
    │         Member states 須指定 notifying authorities
    │
2026.09.11  ★ 漏洞與安全事件通報義務開始適用 ★
    │         （也適用到此前已在 EU 市場上的產品）
    │
2026.12.11  目標：各會員國有足夠的 conformity assessment bodies
    │
    │  ← 36 個月所有要求強制
    │
2027.12.11  ★ CRA 全面強制 ★
              CE Mark 強制要求
              所有義務全面適用
```

---

## 各階段義務與行動對照

### 2026-06-11：Notified Body 規定適用

| 義務 | 影響對象 | 你該做的事 |
|------|---------|-----------|
| Member states 指定 notifying authorities | 所有需第三方評估的產品 | 開始聯繫可能的 notified body |
| Conformity assessment body 制度開始運作 | Class II / 無標準的 Class I | 準備技術文件供評估使用 |

**行動項目：**
- [ ] 確認你的 Class II 產品清單
- [ ] 列出可能的 notified body 候選
- [ ] 準備初步技術文件包

---

### 2026-09-11：漏洞與安全事件通報義務 ★ 最急迫 ★

| 義務 | 詳細內容 | 時限 |
|------|---------|------|
| **Early Warning** | 被積極利用漏洞的早期預警 | **24 小時** |
| **Vulnerability Notice** | 產品資訊、漏洞性質、緩解措施 | **72 小時** |
| **Final Report** | 完整漏洞描述、嚴重性、影響、修復措施 | **14 天** |

**關鍵警告：此通報義務也適用到在 2027-12-11 之前就已放到 EU 市場上的產品。**

**通報對象：** CSIRT / ENISA（透過單一報告平臺）

**行動項目：**
- [ ] 建立 PSIRT 團隊或指定負責人
- [ ] 建立漏洞接收、判級、追蹤流程
- [ ] 建立 24hr / 72hr / 14天 通報 SOP
- [ ] 準備通報模板（Early Warning / Notice / Final Report）
- [ ] 定義內部分工與責任歸屬
- [ ] 對接 CSIRT / ENISA 通報平台
- [ ] 執行至少一次模擬演練

---

### 2027-12-11：CRA 全面強制

| 義務類別 | 具體要求 |
|----------|---------|
| **產品安全要求** | 不存在已知漏洞、默認安全配置、安全更新機制、存取控制、資料加密 |
| **製造商義務** | 設計-開發-生產全程合規、風險評估、技術文件、盡職調查 |
| **漏洞處理** | SBOM、漏洞修復、測試審查、披露政策、更新分發 |
| **通報義務** | 24hr/72hr/14天通報（已於 2026-09-11 生效） |
| **CE 標誌** | 符合性評估通過後貼 CE Mark |
| **技術文件** | 風險評估報告、SBOM、測試報告、DoC |
| **使用者資訊** | 安全手冊、聯繫方式、support end date、更新指引 |
| **Support Period** | 至少 5 年（網通設備通常更長） |
| **文件保存** | 投放市場後至少 10 年或 support period 內（取較長者） |

**行動項目：**
- [ ] 所有 EU 產品完成 SKU 分類
- [ ] 完成符合性評估（Module A / B+C / H）
- [ ] 完成技術文件包
- [ ] 完成使用者資訊文件
- [ ] 準備 EU Declaration of Conformity (DoC)
- [ ] 產品貼 CE 標誌
- [ ] SBOM 完成並持續更新
- [ ] 安全更新機制可運作
- [ ] Support period 政策已公告

## 按產品線的時程規劃

### 資安產品 (Class II) — 最早啟動

| 時段 | 行動 |
|------|------|
| **現在 → 2026.06** | SKU 分類、技術文件骨架、notified body 接洽、PSIRT 建立 |
| **2026.06 → 2026.09** | 通報流程上線、模擬演練、技術文件補齊 |
| **2026.09 → 2027.06** | 正式送 notified body 評估、測試證據補齊 |
| **2027.06 → 2027.12** | 完成評估、DoC、CE 標誌、上市證據包 |

### 網通設備 (Class I) — 第二波

| 時段 | 行動 |
|------|------|
| **現在 → 2026.06** | SKU 分類、support period 政策、LTS 規劃 |
| **2026.06 → 2026.09** | 通報流程上線、SBOM 完成、OTA 驗證 |
| **2026.09 → 2027.06** | 技術文件、風險評估、測試計畫 |
| **2027.06 → 2027.12** | 符合性評估（Module A 或 B+C）、DoC、CE |

### IoT 產品 (逐 SKU 判定) — 第三波

| 時段 | 行動 |
|------|------|
| **現在 → 2026.06** | 邊界盤點、remote data processing 判定、SKU 分級 |
| **2026.06 → 2026.09** | 通報流程上線、App/Cloud SBOM |
| **2026.09 → 2027.06** | 技術文件、第三方元件治理、測試 |
| **2027.06 → 2027.12** | 符合性評估、DoC、CE |

## 已在市場上的產品怎麼辦？

### 通報義務（2026-09-11 起）

- **適用** — 即使產品在 2027-12-11 前已上市，通報義務仍適用
- 意味著：你必須在 2026-09-11 前建好 PSIRT 和通報能力

### 其他義務（2027-12-11 起）

- 在 2027-12-11 前已上市的產品，若之後進行「實質性修改」(substantial modification)，則需符合所有 CRA 義務
- 純安全更新（security patches）不構成實質性修改
- 重大功能更新可能構成實質性修改

---

## 關鍵判定提醒

1. **Class II 必須第三方評估** — 不能只靠自我宣告
2. **Class I 不一定能自我評估** — 需要有適用的 harmonised standard
3. **Support period 不是固定 5 年** — 網通設備通常要更長
4. **IoT 不等於 Class I** — 要看 core functionality
5. **Cloud/App 可能在 CRA 邊界內** — 看是否為核心功能必要
6. **通報義務先於全面適用** — 2026-09-11 vs 2027-12-11
7. **通報義務追溯既有產品** — 不能等到 2027 才準備
